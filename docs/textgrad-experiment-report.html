실험 설계

학습 노드 (파라미터) = 설명 프롬프트 1개
입력(미니배치) = 개념 5개: 재귀, 복리, 무지개, 광합성, 베이즈 정리
초기 프롬프트(결함): "주어진 개념을 설명하라."
Loss = 위반 규칙 수(0–3), 5개 평균 · 100% 프로그램 채점:
  • R1 60단어 이하
  • R2 비유 표현 포함 (처럼/마치/비유하자면…)
  • R3 마지막 줄이 요약:으로 시작
Gradient 정의 (우리식 "semantic gradient"): 막연한 "개선"이 아니라 — (a) 어떤 규칙이 위반됐는지 특정, (b) 만족시키려면 프롬프트가 무엇을 추가로 지시해야 하는지 방향. 실제 위반된 규칙만, 최소한으로.
optimizer.step(): gradient를 반영하되 최소 편집 + 길이 정칙화(3문장 이내·개념 비종속) — 일반해로 수렴하도록.

관찰 — 개념적 목표와의 연결

1gradient가 의미적으로 "특정"됐다

매 스텝 "막연히 더 좋게"가 아니라 R1/R2/R3 중 무엇이·왜 위반됐고 어떤 지시를 더해야 하는지를 짚었다. → semantic gradient = 방향 + 어긴 규칙 특정이라는 정의가 실제로 작동.

2단순 규칙 추가를 넘어선 credit assignment (핵심)

iter 1→2가 결정적이다: 프롬프트는 이미 "요약: 줄을 추가하라"고 지시하고 있었는데도 실패했다. gradient는 이를 보고 "지시는 있으나 강제성이 부족하다 → 의무 표현으로 강화하라"고 진단했다. 즉 "무엇을 빠뜨렸나"가 아니라 "왜 여전히 안 되나(영향력)"를 파악해 교정 — 우리가 원한 "결과에 대한 영향력 파악 후 교정"의 1차 증거.

3길이 정칙화가 유지돼 일반해로 수렴

최종 프롬프트가 여전히 한두 문장·개념 비종속(일반). 사례별 패치로 비대해지지 않고 일반 규칙으로 수렴 — 기존 문헌이 안 다루는 우리 차별점(길이/MDL 정칙화)이 의도대로 작동.
